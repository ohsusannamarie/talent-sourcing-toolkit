# GitHub Sourcing Guide

GitHub sourcing is not just keyword searching profiles.

It is reading public technical signals: repositories, contributions, projects, languages, docs, issues, stars, forks, communities, and the way engineers show their work.

---

## What GitHub can help recruiters find

- Engineers with public project work
- Open-source contributors
- People using specific languages or frameworks
- Maintainers of relevant tools
- Conference speakers and technical writers
- Contributors to company or community projects
- Adjacent talent pools outside LinkedIn-heavy searches

---

## Search targets

### Profiles

Useful for identifying people and basic public signals.

Look for:

- Name
- Location
- Bio keywords
- Company
- Website
- Social links
- Pinned repositories

### Repositories

Useful for understanding what someone builds or contributes to.

Look for:

- Languages
- Topics
- README quality
- Recent activity
- Stars and forks
- Project relevance
- Contribution history

### Code and documentation

Useful for finding people working with specific tools, libraries, APIs, or technologies.

Look for:

- Technology keywords
- Package names
- Config files
- Framework references
- Documentation contributions
- Issue and pull request participation

---

## Starter workflow

1. Translate the role into technical keywords.
2. Identify languages, frameworks, tools, and adjacent terms.
3. Search repositories and code before profiles.
4. Review contributor lists, commits, issues, and maintainers.
5. Cross-reference public profile links carefully.
6. Validate fit with multiple signals, not one keyword match.
7. Personalize outreach based on actual public work.

---

## Outreach angle

Bad GitHub outreach says:

> I saw your GitHub and thought you would be a fit.

Better GitHub outreach says:

> I came across your work on [project/topic] and noticed your experience with [specific technology]. I am working on a role where that kind of background is directly relevant.

Specific beats creepy. Always.

---

## Practical rule

Do not treat GitHub as a resume database.

Treat it as public technical context, then use judgment.
